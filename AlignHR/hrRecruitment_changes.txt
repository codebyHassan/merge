# Recruitment Workflow Refactor Review

## Root Cause

The current workflow routes approvals using RoleId.

Current design:

```text
HR_WorkflowStep
---------------
RoleId
```

Current behavior:

```text
Step 1 = HRBP Role
Step 2 = HOD Role
Step 3 = Recruiter Role
```

Inbox logic then retrieves pending requests by matching user roles.

Example:

```csharp
step.RoleId == currentUserRoleId
```

This causes:

* All HRBPs to see all HRBP approvals.
* All HODs to see all HOD approvals.
* No department-level ownership.
* Incorrect approval visibility.
* Poor scalability for enterprise organizations.

This is the primary architectural flaw.

---

# Remove / Refactor

## Remove dependency on RoleId for approval execution

Workflow routing should never determine approvers directly from user roles.

Keep roles only for permissions:

Examples:

```text
CanCreateRequisition
CanApproveRequisition
CanManageRecruitment
```

Do not use roles to determine who receives a workflow task.

# New Submission Logic

When requisition is submitted:

1. Read DepartmentId.
2. Load WorkflowDefinition.
3. Load WorkflowSteps.
4. Resolve each ApproverType from HR_DepartmentApprover.
5. Create WorkflowInstanceStep records.
6. Mark first step Pending.
7. Mark remaining steps Waiting.

---

# New Inbox Logic

Current:

```csharp
RoleId == CurrentUserRoleId
```

Replace with:

```csharp
EmployeeId == CurrentEmployeeId
```

Only assigned approvers should see requests.

---

# What To Delete

Delete workflow routing based on:

```text
RoleId
UserRoles
Role Matching
```

for approval assignment.

Keep RBAC only for authorization.

---

# Expected Result

IT Requisition:

```text
HRBP      Sara
HOD       Ahmed
Recruiter Ali
```

Only Sara sees first approval.

After approval:

Only Ahmed sees second approval.

After approval:

Only Ali sees recruiter assignment.

No other department users can see or act on the request.
