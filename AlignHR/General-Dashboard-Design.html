Design Recommendation · General Dashboard

One home screen that works for every user — no matter which modules are on.

AlignHR ships 16 modules, and each one carries its own dashboard. A client who enables only two or three leaves most users staring at empty space on login. This is a module-aware home built on what every user always has: an identity, a company to hear from, and people to reach.

16 modules mapped 9 universal widgets Permission-gated rendering Bootstrap 5 grid Responsive · light + dark
The gap

Every module has a dashboard. Nothing owns the front door.

Because modules are toggled per client, the login landing has to assume nothing is present. Instead of gating the whole home behind optional data, the General Dashboard is built from the three things that survive every configuration — and then invites module widgets in as they're enabled.

Three constants behind the design

Identity — everyone who logs in is an employee with a profile, role, department, and manager (Core HR is foundational, never optional).
Communication — announcements surface across ESS, MSS, and Engagement; broadcasting to staff doesn't depend on any one module.
Reference & people — holidays, policies, HR/IT contacts, and the staff directory are company data every user can use today.

Reading this document

Solid card = Universal

Always renders. Depends only on identity, communication, or reference data — present in every client configuration.

Dashed card = Module-gated

Renders only when its module is enabled and the user has permission. Absent modules leave no hole — the grid reflows.

Deliverable · Integration strategy

Letting modules in without crowding the room

Five rules keep the dashboard full-but-calm as modules are switched on. They lean on infrastructure AlignHR already has — the session's menu and permission set.

01

Gate on access, not guesswork

Render a module widget only when the user can reach that area — reuse UserSession.MenuItems / HasControllerAccess(). That single check already encodes both client enablement and the user's role.

02

One card contract for all

Every widget is the same Bootstrap card: header (icon + title + View all), a body capped at 3–5 rows, and a built-in empty state. Uniform shape means any mix still looks composed.

03

Action before information

Widgets sort by urgency: approvals and pending tasks float to the top of the column; read-only summaries settle below. Attention lands where it's needed first.

04

Reflow, never reserve

Disabled modules emit no markup, so the responsive grid closes the gap — no placeholders, no dead space. A two-module tenant and a twelve-module tenant both look intentional.

05

Summarize here, act there

The dashboard shows the headline and links out; it never rebuilds a module. View all → hands off to the module's own screen, keeping the home light and load times fast.

06

Let users pin (phase 2)

A Customize affordance to show/hide and reorder cards, persisted per user. Ships after the fixed-priority layout proves out — modern SaaS expectation, not a launch blocker.

Hierarchy, top to bottom

1 · Orient — greeting + identity.  •  2 · Signal — KPI strip (universal metrics; module metrics slot in when present).  •  3 · Act — Quick Actions launcher.  •  4 · Absorb — announcements + module summaries in the wide column.  •  5 · Reference — profile, links, holidays, directory in the rail.   Attention flows from who am I → what needs me → what can I do → what's happening → what can I look up.

AlignHR General Dashboard — design recommendation. Universal widgets derive from identity, communication, and reference data; module widgets render through the existing session permission model. Built for Bootstrap 5, responsive across desktop / tablet / mobile, light and dark themes.
