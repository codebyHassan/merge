BUGS FOUND AND FIXED BY CLAUDE — SPRINT 2 (2026-06-05)
================================================================

BUG 1 - FIXED: SubmitAsync never started the workflow (critical logic error)
  Files: Services/IHrRequisitionService.cs
         Services/HrRequisitionService.cs
         Controllers/HrRequisitionsController.cs
  Problem: When a user saves a Draft and then clicks Submit from the Details view,
           the Submit action calls SubmitAsync. SubmitAsync only set Status="Submitted"
           and saved — it never called _workflowService.StartAsync(). The workflow
           instance was never created. The requisition would be stuck as "Submitted"
           with no approvers in the queue, no history, and no WorkflowInstanceFK.
           The Create/Edit POST paths DO start the workflow correctly.
           Only the Details-view Submit path was broken.
  Fix: Added decimal? actionBy parameter to IHrRequisitionService.SubmitAsync.
       SubmitAsync now resolves actionBy (from session or requisition.EmployeeFK fallback)
       and calls _workflowService.StartAsync() before saving.
       StartAsync sets requisition.Status="Submitted" and WorkflowInstanceFK.
       Controller Submit action now passes GetActionByEmployeeId().

BUG 2 - FIXED: Sidebar had no "Approval Inbox" navigation link
  File: Views/Shared/_Sidebar.cshtml
  Problem: Approvers (HRBP, HOD, Recruiter roles) had no way to reach the Inbox
           screen from the sidebar navigation.
  Fix: Added "Approval Inbox" sublink under the Recruitment menu.
       Wired active state to "HrRequisitionsInbox" page key.
       Updated Inbox.cshtml and Action.cshtml to use ViewBag.ActivePage = "HrRequisitionsInbox"
       so both screens highlight the Inbox link and keep the menu expanded.
       Updated recruitmentOpen to include "HrRequisitionsInbox" key.

SPRINT 7 — ONBOARDING & EMPLOYEE CONVERSION (2026-06-05)
================================================================

Sprint 7 goal: After a candidate is Hired, start pre-onboarding, collect documents,
run IT/Admin/HR task checklists, confirm the candidate has physically joined, and
then automatically create an Employee record and User account.

Key architecture rule (from spec):
  DO NOT create an Employee record when the offer is accepted.
  Create it ONLY when candidate status = Joined (joining physically confirmed).
  Prevents ghost employees, no-shows, and accepted-but-never-joined cases.

Business flow added:
  Hired → Start Onboarding → Tasks Assigned → Documents Collected
        → Confirm Joining → Employee Created → User Account Created → Role Assigned

Tables created (all NUMERIC(18,0) PKs, via migration 20260605060000):
  HR_OnboardingTaskTemplate — reusable task definitions per department
                              seeded with 8 defaults: Create Email Account, Allocate Laptop,
                              Issue Access Card, Assign Buddy, Prepare Workstation,
                              Payroll Registration, Add to Leave System, Safety Induction
  HR_Onboarding             — main onboarding record per offer; unique index on OfferId
                              Status lifecycle: Initiated → DocumentsPending → TasksInProgress
                                               → ReadyToJoin → Joined → Completed / Cancelled
  HR_OnboardingTask         — actual task instances (seeded from all active templates on start)
                              Status: Pending / InProgress / Completed / Cancelled
  HR_OnboardingDocument     — documents submitted by candidate (CNIC, Passport, Degree, etc.)
                              stores file path only — no binary in SQL
  HR_JoiningConfirmation    — trigger for employee creation; unique index on OnboardingId
                              (one confirmation per onboarding)

      1. Loads HrCandidate (FirstName, LastName, Email)

      3. Creates Employee record (Code, FirstName, LastName, DepartmentFk, LocationFk,

      6. Returns new Employee.Id
